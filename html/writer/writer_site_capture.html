Site Capture Plan — Four-Format Parquet vs InfluxDB2

Real-site data capture test  ·  Long-format parquet baseline + InfluxDB2.0 concurrent capture  ·  Post-capture: 4 format conversions + validation

Overview

This test captures one day of live site data in parallel to both long-format parquet (via parquet_writer) and InfluxDB2.0 (via the existing on-site Telegraf chain). After the capture window, the raw long-format parquet is converted to four formats and all four are validated against the InfluxDB ground truth.

Phase 1 — On-Site Capture Config

Topic segment mapping

The MQTT topic published by the site follows the same structure as the bench. The writer's topic_segments names become parquet column names, so we rename here to match the target schema exactly:

Topic position Bench segment name Site segment name Action Example value
0 source_type source_type Drop bench
1 unit_id unit_id Keep → column + partition key 0215D1D8
2 device (category) device_type Keep → column bms
3 instance (specific) device Keep → column bms_1
4 point_name point_name Keep → column SysSOC
5 dtype_hint dtype_hint Drop float
Column order note: Parquet stores columns in declaration order, but DuckDB and PyArrow queries are column-name-based. The schema (ts, value, device_type, device, point_name, unit_id) is equivalent to the manager's (ts, value, device, device_type, point_name, unit_id) as long as column names match. The writer emits them in the order they are encountered; DuckDB SELECT ts, value, device, device_type, point_name, unit_id returns them in any desired order regardless.

site_capture_long.yaml

This config produces the raw long-format parquet. Key differences from the bench wide config: compound_field_name is absent (disables wide-pivot), time_window_ms is absent (each MQTT message = one row), flush is 10 minutes, compaction is off during capture.

Phase 3 — Four Output Formats

All four formats are produced from the same raw long-format parquet by long_to_wide.py. The script uses DuckDB for all transformations.

A Wide format — data dedupe null_fill_unchanged

Long rows are pivoted to wide (one row per timestamp per unit, one column per signal path device_type/device/point_name). Within each flush window, any signal whose value has not changed since the previous flush is stored as NULL. This is equivalent to enabling null_fill_unchanged: true in the writer.

Expected result: ~78–85% nulls across all rows. Constant signals (SOH = 100.0 over many hours) compress to ~55 bytes per column. Dynamic signals retain every value.

-- Step 1: build the wide pivot with last-value-per-window logic
-- Signal column name format: device_type || '/' || device || '/' || point_name
CREATE TABLE wide_raw AS
PIVOT (
  SELECT ts, unit_id,
         device_type || '/' || device || '/' || point_name AS signal,
         value
  FROM read_parquet('raw/**/*.parquet', union_by_name=true)
)
ON signal USING LAST(value) GROUP BY ts, unit_id ORDER BY unit_id, ts;

-- Step 2: null out values that match the previous row for the same unit
-- (simulates null_fill_unchanged — signals only written when they change)
CREATE TABLE wide_deduped AS
SELECT ts, unit_id,
  CASE WHEN sig = LAG(sig) OVER (PARTITION BY unit_id ORDER BY ts) THEN NULL ELSE sig END AS sig
  -- ... repeated for each signal column (generated dynamically by script)
FROM wide_raw;

Output dir: out/A_wide_deduped/

B Wide format — deduped data reinserted forward-fill

Same pivot as Format A, but all NULL cells are forward-filled to restore the last known value. Every row has a complete set of signal values with no nulls. Represents what a query would return after applying COALESCE(val, last_val).

Expected result: 0% nulls. Largest file size of the four formats. Compresses worst (forward-filled constants repeat identically, so RLE still helps, but the explicit values increase the column chunk size vs true null-fill).

-- Start from Format A (wide_deduped), forward-fill each signal column
CREATE TABLE wide_filled AS
SELECT ts, unit_id,
  LAST_VALUE(sig IGNORE NULLS) OVER (
    PARTITION BY unit_id ORDER BY ts
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS sig
  -- ... repeated for each signal column
FROM wide_deduped;

Output dir: out/B_wide_filled/

C Long format — data dedupe (1 write per minute) resampled

Long schema kept. Rows are binned to 1-minute intervals; within each bin the LAST value is taken for each (unit_id, device_type, device, point_name) combination. For 1 Hz data this reduces row count by ~60×. Equivalent to what the on-site InfluxDB capture does when telegraf sends one value per minute (deduped at source).

Expected result: ~1/60 the rows of Format D. Bytes/signal-value similar to Format D but with far fewer rows.

-- Bin to 1-minute buckets, keep last value per signal
CREATE TABLE long_deduped AS
SELECT
  TIME_BUCKET(INTERVAL '1 minute', ts) AS ts,
  unit_id, device_type, device, point_name,
  LAST(value ORDER BY ts) AS value
FROM read_parquet('raw/**/*.parquet', union_by_name=true)
GROUP BY 1, 2, 3, 4, 5
ORDER BY unit_id, ts;

-- DuckDB alternative if TIME_BUCKET not available:
  epoch_ms(ts) / 60000 * 60000 AS ts_bin   -- floor to minute boundary

Output dir: out/C_long_deduped/

D Long format — un-deduped (raw as captured) raw

The original raw long-format parquet as written by the writer. Every MQTT message is one row. No transformation. This is the reference input for all other formats.

Expected result: one row per signal per second (1 Hz), 63 signals, N units. For a 24-hour run: 24 × 3600 × 63 × N_units rows.

-- No transformation; just re-partition for consistency
-- (or symlink raw/ to out/D_long_raw/)
COPY (SELECT * FROM read_parquet('raw/**/*.parquet', union_by_name=true)
      ORDER BY unit_id, ts)
TO 'out/D_long_raw/' (FORMAT parquet, PARTITION_BY (unit_id), COMPRESSION snappy);

Output dir: out/D_long_raw/

EKS Container Deployment

parquet_writer runs as a single-container Deployment in EKS. The writer config is injected via a ConfigMap, output parquet lands on an EFS-backed PersistentVolume (shared ReadWriteMany so post-processing Jobs can read the same mount), and the health endpoint is exposed as a liveness probe. MQTT broker address is passed as an environment variable so the same image works across sites.

Architecture overview

ComponentK8s resourceNotes
Writer config ConfigMap parquet-writer-config Mounted as /etc/writer/config.yaml inside container
InfluxDB token / credentials Secret parquet-writer-secrets Env vars INFLUX_TOKEN, MQTT_HOST, SITE_ID
Parquet output PVC → EFS StorageClass ReadWriteMany so conversion Job mounts the same volume after capture
Writer process Deployment (1 replica) Liveness probe on :8771/health; restartPolicy Always
Post-processing Job long-to-wide Run manually after capture; mounts same PVC; writes 4 format subdirs
Validation Job validate-formats Mounts PVC + reaches InfluxDB; writes HTML report to PVC
EFS vs EBS: use EFS (NFS) not EBS. The writer Deployment and the post-processing Job both need to mount the same volume. EBS is ReadWriteOnce (single node); EFS is ReadWriteMany and works across nodes. If EFS is not available, use S3 sync after capture instead (see below).

# pvc-parquet-capture.yaml

Container entrypoint

A small shell script replaces the placeholder strings in the ConfigMap copy before launching the writer. This keeps the ConfigMap site-agnostic. 

#!/bin/sh
# entrypoint.sh — baked into the container image
set -e
cp /etc/writer/config.yaml /tmp/config.yaml
sed -i "s/MQTT_HOST_PLACEHOLDER/${MQTT_HOST}/g" /tmp/config.yaml
sed -i "s/SITE_ID_PLACEHOLDER/${SITE_ID}/g"   /tmp/config.yaml
exec /usr/local/bin/parquet_writer --config /tmp/config.yaml

Phase 4 — Validation Against InfluxDB2.0

validate_formats.py fetches the InfluxDB data for the exact capture window (using influx_to_parquet.py as a library), then runs DuckDB comparison queries against each of the four parquet formats.

Validation checks per format

Check Query approach Pass criterion
Signal coverage SELECT DISTINCT unit_id, device_type, device, point_name from parquet vs InfluxDB tags Same set of (unit_id, point_name) tuples in both
Time range MIN(ts) and MAX(ts) from parquet vs InfluxDB first() / last() Both within 1 flush-interval (10 min) of T_START / T_STOP
Row count per signal COUNT(*) GROUP BY unit_id, point_name in parquet vs InfluxDB count() Format D ≅ InfluxDB (±1 per flush boundary).
Format C ≅ InfluxDB/60.
Formats A/B: rows = unique timestamps (always ≤ Format D)
Value accuracy — spot check Sample 100 random (ts, unit_id, point_name) triples from InfluxDB. Look up same key in each parquet format. Compare values. Absolute difference < 1e−6 (float precision)
Value accuracy — aggregate AVG(value), STDDEV(value), MIN(value), MAX(value) per (unit_id, point_name) in parquet vs InfluxDB AVG within 0.01%, min/max identical
No phantom signals Signals present in parquet but absent in InfluxDB Set difference = empty
No dropped signals Signals present in InfluxDB but absent in parquet Set difference = empty
Bytes / signal-value file_size / (row_count × signal_count) per format vs influx_bytes / influx_row_count Parquet formats < InfluxDB for all formats (expected)

Key validation queries

-- Signal coverage: find any signal in InfluxDB not in parquet format D
SELECT influx.unit_id, influx.point_name
FROM influx_signals influx
LEFT JOIN (
  SELECT DISTINCT unit_id, point_name
  FROM read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true)
) pq USING (unit_id, point_name)
WHERE pq.unit_id IS NULL;

-- Row count agreement (Format D vs InfluxDB, per signal)
SELECT
  p.unit_id, p.point_name,
  p.pq_rows, i.inf_rows,
  ROUND(100.0 * ABS(p.pq_rows - i.inf_rows) / i.inf_rows, 2) AS diff_pct
FROM (
  SELECT unit_id, point_name, COUNT(*) AS pq_rows
  FROM read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true)
  GROUP BY 1, 2
) p
JOIN (
  SELECT unit_id, point_name, inf_count AS inf_rows FROM influx_counts
) i USING (unit_id, point_name)
ORDER BY diff_pct DESC;

-- Value accuracy: spot-check 100 random samples from InfluxDB against Format D
SELECT
  s.ts, s.unit_id, s.point_name,
  s.influx_value,
  p.value AS pq_value,
  ABS(s.influx_value - p.value) AS delta
FROM influx_sample s
JOIN read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true) p
  ON p.ts = s.ts AND p.unit_id = s.unit_id AND p.point_name = s.point_name
WHERE delta > 1e-6
ORDER BY delta DESC;

long_to_wide.py — design spec

Usage:
  python3 long_to_wide.py --input raw/ --output out/ [--units 0215D1D8,...]

Arguments:
  --input   dir containing raw long-format parquet (all units, all days)
  --output  output root; subdirs A_wide_deduped/ B_wide_filled/ C_long_deduped/ D_long_raw/
  --units   optional comma-separated unit_id filter
  --start   optional start ISO timestamp
  --stop    optional stop ISO timestamp

Algorithm:
  1. Load raw via DuckDB read_parquet with union_by_name=true
  2. Build signal list: SELECT DISTINCT device_type||'/'||device||'/'||point_name
  3. Generate dynamic PIVOT SQL over signal list  (DuckDB PIVOT clause)
  4. Format D: COPY raw to out/D with partition_by unit_id
  5. Format C: bin to 1-min, LAST(value), COPY to out/C
  6. Format A: PIVOT → wide_raw, then lag-compare to null unchanged values
  7. Format B: COPY wide_raw to out/B (all values; no null-fill)
  8. Print stats table: rows / files / bytes / bytes-per-signal-value for each format

validate_formats.py — design spec

Usage:
  python3 validate_formats.py \
    --influx-url http://site-host:8086 --token TOKEN --org ORG --bucket BUCKET \
    --start 2026-04-14T09:00:00Z --stop 2026-04-15T09:00:00Z \
    --output out/ \
    --report validation_report.html

Algorithm:
  1. Fetch InfluxDB data for window via influx_to_parquet.fetch_measurement()
     → saves as influx_ground_truth.parquet
  2. For each of the 4 parquet output dirs:
     a. Signal coverage check (set intersection / difference vs InfluxDB signals)
     b. Time range check (min/max ts alignment)
     c. Row count per signal (vs InfluxDB count; expected ratios for each format)
     d. Value spot-check (100 random samples from InfluxDB → lookup in parquet)
     e. Value aggregate check (AVG, MIN, MAX per signal vs InfluxDB)
     f. Bytes/signal-value calculation
  3. Write HTML report: pass/fail table per format per check
     + per-format size/rows summary + bytes/signal-value bar chart
