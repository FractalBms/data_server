1. Overview

The build script (scripts/evelyn-k8s-build.sh) runs entirely on the developer machine. It uses ephemeral Docker containers to compile the C++ binaries so the host needs no Arrow/Parquet dev headers. Everything produced — binaries, configs, HTML pages, Python and shell scripts, and Kubernetes manifests — is bundled into a single evelyn-deploy-<date>.tar.gz.

What the script doesTool usedOutput
Compile parquet_writerDocker (ubuntu:24.04)bin/parquet_writer
Compile flashmqDocker (ubuntu:24.04, builds from source)bin/flashmq

No pre-installed Arrow/Parquet, mosquitto, or simdjson dev headers are required on the host. Docker pulls a clean Ubuntu 24.04 image and installs everything inside the build container.

3. Build Script: evelyn-k8s-build.sh

scripts/evelyn-k8s-build.sh
#!/usr/bin/env bash
# evelyn-k8s-build.sh — build all Evelyn components and produce a deployment tar
#
# Usage:  bash scripts/evelyn-k8s-build.sh [--tag evelyn:1.2.3]
#
# Requires: docker (running), git (for FlashMQ clone inside container)
# Output:   dist/evelyn-deploy-YYYYMMDD.tar.gz
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
DATE="$(date +%Y%m%d)"
OUT_NAME="evelyn-deploy-${DATE}.tar.gz"
OUT="$REPO_ROOT/dist/$OUT_NAME"
BUILD_DIR="$(mktemp -d /tmp/evelyn-build-XXXXXX)"
STAGE="$BUILD_DIR/evelyn-deploy"
FLASHMQ_TAG="v1.9.1"
UBUNTU_IMAGE="ubuntu:24.04"

trap 'echo "[build] cleaning up $BUILD_DIR"; rm -rf "$BUILD_DIR"' EXIT

echo "=== Evelyn Kubernetes build ==="
echo "    repo:   $REPO_ROOT"
echo "    output: $OUT"
echo "    stage:  $STAGE"
echo

# ── Prerequisite checks ───────────────────────────────────────────────────────
if ! command -v docker &>/dev/null; then
    echo "ERROR: docker not found — install Docker and ensure the daemon is running"
    exit 1
fi
if ! docker info &>/dev/null; then
    echo "ERROR: Docker daemon not running or permission denied"
    exit 1
fi

mkdir -p "$REPO_ROOT/dist"
mkdir -p \
    "$STAGE/bin" \
    "$STAGE/html/writer" \
    "$STAGE/configs" \
    "$STAGE/scripts" \
    "$STAGE/k8s"

# ── Step 1: build parquet_writer ─────────────────────────────────────────────
# (defined in its own section below)
build_parquet_writer

# ── Step 2: build FlashMQ ────────────────────────────────────────────────────
build_flashmq

# ── Step 3: collect repo files ────────────────────────────────────────────────
collect_files

# ── Step 4: write Kubernetes manifests ────────────────────────────────────────
write_k8s_manifests

# ── Step 5: package ───────────────────────────────────────────────────────────
tar -czf "$OUT" -C "$BUILD_DIR" evelyn-deploy/
echo
echo "=== Done ==="
echo "    $OUT"
echo "    $(du -sh "$OUT" | cut -f1)  $(tar -tzf "$OUT" | wc -l) files"

The five build functions are defined below. They are all in the same script file — split here for clarity.

4. Step 1 — Build parquet_writer (Docker)

Mounts source/parquet_writer_cpp/ read-only into an Ubuntu 24.04 container, installs dev headers, and runs make parquet_writer. The compiled binary is written to the host via a second /out mount.

build_parquet_writer() {
    echo "[step 1] building parquet_writer..."

    # /src is read-only so the container cannot dirty the repo.
    # /out is writable — the binary lands at $BUILD_DIR/parquet_writer.
    docker run --rm \
        -v "$REPO_ROOT/source/parquet_writer_cpp":/src:ro \
        -v "$BUILD_DIR":/out \
        "$UBUNTU_IMAGE" bash -c "
            set -e
            export DEBIAN_FRONTEND=noninteractive
            apt-get update -q
            apt-get install -y -q --no-install-recommends \
                build-essential \
                libarrow-dev libparquet-dev \
                libmosquitto-dev libyaml-cpp-dev \
                libsnappy-dev libzstd-dev liblz4-dev \
                libssl-dev libsimdjson-dev
            cp -r /src /build && cd /build
            make parquet_writer
            cp parquet_writer /out/parquet_writer
        "

    cp "$BUILD_DIR/parquet_writer" "$STAGE/bin/parquet_writer"
    chmod +x "$STAGE/bin/parquet_writer"
    echo "    → bin/parquet_writer  ($(du -sh "$STAGE/bin/parquet_writer" | cut -f1))"
}
Why copy /src to /build? Arrow's pkg-config sometimes resolves relative paths. Copying to /build inside the container avoids any path issues with the read-only mount and keeps the host repo clean.

5. Step 2 — Build FlashMQ (Docker)

Clones the tagged FlashMQ release inside an Ubuntu 24.04 container and builds with CMake. No source lives in the repo — the clone happens entirely inside the container.

build_flashmq() {
    echo "[step 2] building FlashMQ $FLASHMQ_TAG..."

    docker run --rm \
        -v "$BUILD_DIR":/out \
        "$UBUNTU_IMAGE" bash -c "
            set -e
            export DEBIAN_FRONTEND=noninteractive
            apt-get update -q
            apt-get install -y -q --no-install-recommends \
                cmake build-essential git libssl-dev
            git clone --branch $FLASHMQ_TAG --depth 1 \
                https://github.com/halfgaar/FlashMQ.git /flashmq
            cmake -B /flashmq/build -S /flashmq \
                  -DCMAKE_BUILD_TYPE=Release
            cmake --build /flashmq/build --parallel \$(nproc)
            cp /flashmq/build/flashmq /out/flashmq
        "

    cp "$BUILD_DIR/flashmq" "$STAGE/bin/flashmq"
    chmod +x "$STAGE/bin/flashmq"
    echo "    → bin/flashmq  ($(du -sh "$STAGE/bin/flashmq" | cut -f1))"
}
Internet required during build. The FlashMQ clone fetches from GitHub. For air-gapped environments, pre-clone FlashMQ into docker/evelyn-container/flashmq-src/ and mount it into the container instead of cloning.

6. Step 3 — Collect repo files

All HTML pages, Python scripts, shell scripts, config files, and the Option C Dockerfile are copied from the live repo tree into the staging directory.

collect_files() {
    echo "[step 3] collecting repo files..."
    R="$REPO_ROOT"
    S="$STAGE"

    # ── HTML pages ────────────────────────────────────────────────────────────
    cp "$R/html/writer/evelyn_docker_34.html"         "$S/html/index.html"
    cp "$R/html/parquet_monitor.html"                  "$S/html/"
    cp "$R/html/writer/parquet_viewer.html"            "$S/html/writer/"
    cp "$R/html/writer/writer_design.html"             "$S/html/writer/"
    cp "$R/html/writer/evelyn_real_time_data.html"     "$S/html/writer/"
    cp "$R/html/writer/evelyn_container_design.html"   "$S/html/writer/"
    cp "$R/html/writer/evelyn_writer_project_build.html" "$S/html/writer/"

    # ── Python scripts ────────────────────────────────────────────────────────
    # bridge_api_container.py is the supervisord-aware management API (Option C).
    # bridge_api_native.py is the systemd-aware version for bare-metal installs.
    cp "$R/docker/evelyn-container/bridge_api_container.py"  "$S/"
    cp "$R/docker/evelyn-native/bridge_api_native.py"        "$S/"

    # ── Shell scripts ─────────────────────────────────────────────────────────
    cp "$R/docker/evelyn-native/install.sh"           "$S/scripts/"
    cp "$R/scripts/gx10-evelyn-start.sh"              "$S/scripts/"
    cp "$R/scripts/gx10-evelyn-stop.sh"               "$S/scripts/"
    cp "$R/scripts/gx10-evelyn-4sim-start.sh"         "$S/scripts/"
    cp "$R/scripts/gx10-evelyn-4sim-stop.sh"          "$S/scripts/"
    cp "$R/scripts/evelyn-k8s-build.sh"               "$S/scripts/"  # this script
    chmod +x "$S/scripts/"*.sh

    # ── Configs ───────────────────────────────────────────────────────────────
    # These are the defaults baked into the container.
    # entrypoint.sh installs them to /etc/evelyn/ on first run if not present.
    cp "$R/docker/evelyn-native/configs/flashmq.conf"   "$S/configs/flashmq.conf"
    cp "$R/docker/evelyn-container/configs/writer.yaml"  "$S/configs/writer.yaml"

    # ── Option C container files ──────────────────────────────────────────────
    cp "$R/docker/evelyn-container/Dockerfile"          "$S/Dockerfile"
    cp "$R/docker/evelyn-container/supervisord.conf"     "$S/supervisord.conf"
    cp "$R/docker/evelyn-container/entrypoint.sh"        "$S/entrypoint.sh"
    chmod +x "$S/entrypoint.sh"

    echo "    → $(find "$S" -type f | wc -l) files staged"
}
All shell scripts are chmod +x in the staging directory so they are executable when extracted on the target machine without needing a separate chmod step.

7. Step 4 — Write Kubernetes manifests

The K8s manifests are written as heredocs inside the build script — no separate template files to maintain. They are generated fresh each build so the image tag embedded in deployment.yaml is always current.

write_k8s_manifests() {
    echo "[step 4] writing Kubernetes manifests..."
    K="$STAGE/k8s"
    IMAGE_TAG="${EVELYN_IMAGE:-evelyn:latest}"

    # namespace.yaml
    cat > "$K/namespace.yaml" <<'EOF'
apiVersion: v1
kind: Namespace
metadata:
  name: evelyn
EOF

    # configmap.yaml  — default configs; override with site-specific values
    cat > "$K/configmap.yaml" <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: evelyn-config
  namespace: evelyn
data:
  flashmq.conf: |
$(sed 's/^/    /' "$STAGE/configs/flashmq.conf")
  writer.yaml: |
$(sed 's/^/    /' "$STAGE/configs/writer.yaml")
EOF

    # pvc.yaml
    cat > "$K/pvc.yaml" <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: evelyn-parquet-pvc
  namespace: evelyn
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  # storageClassName: ""   # uncomment and set for a specific StorageClass
EOF

    # deployment.yaml
    cat > "$K/deployment.yaml" <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: evelyn
  namespace: evelyn
  labels:
    app: evelyn
spec:
  replicas: 1
  selector:
    matchLabels:
      app: evelyn
  template:
    metadata:
      labels:
        app: evelyn
    spec:
      containers:
        - name: evelyn
          image: ${IMAGE_TAG}
          imagePullPolicy: IfNotPresent
          ports:
            - name: mqtt
              containerPort: 1883
              protocol: TCP
            - name: mqtt-ws
              containerPort: 9001
              protocol: TCP
            - name: http-ui
              containerPort: 8080
              protocol: TCP
            - name: health
              containerPort: 8771
              protocol: TCP
            - name: bridge-api
              containerPort: 8772
              protocol: TCP
          volumeMounts:
            - name: parquet-data
              mountPath: /data/parquet
            - name: evelyn-config
              mountPath: /etc/evelyn
              readOnly: false
          readinessProbe:
            httpGet:
              path: /health
              port: 8771
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8771
            initialDelaySeconds: 15
            periodSeconds: 30
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              cpu: "4"
              memory: "4Gi"
      initContainers:
        - name: init-config
          image: busybox:1.36
          command:
            - sh
            - -c
            - |
              if [ ! -f /etc/evelyn/writer.yaml ]; then
                echo "[init] installing default writer.yaml"
                cp /etc/evelyn-defaults/writer.yaml /etc/evelyn/writer.yaml
              fi
              if [ ! -f /etc/evelyn/flashmq.conf ]; then
                echo "[init] installing default flashmq.conf"
                cp /etc/evelyn-defaults/flashmq.conf /etc/evelyn/flashmq.conf
              fi
          volumeMounts:
            - name: evelyn-config
              mountPath: /etc/evelyn
            - name: evelyn-defaults
              mountPath: /etc/evelyn-defaults
              readOnly: true
      volumes:
        - name: parquet-data
          persistentVolumeClaim:
            claimName: evelyn-parquet-pvc
        - name: evelyn-config
          emptyDir: {}
        - name: evelyn-defaults
          configMap:
            name: evelyn-config
EOF

    # service.yaml
    cat > "$K/service.yaml" <<'EOF'
apiVersion: v1
kind: Service
metadata:
  name: evelyn
  namespace: evelyn
spec:
  type: NodePort
  selector:
    app: evelyn
  ports:
    - name: mqtt
      port: 1883
      targetPort: 1883
      nodePort: 31883
      protocol: TCP
    - name: mqtt-ws
      port: 9001
      targetPort: 9001
      nodePort: 31901
      protocol: TCP
    - name: http-ui
      port: 8080
      targetPort: 8080
      nodePort: 30880
      protocol: TCP
    - name: health
      port: 8771
      targetPort: 8771
      nodePort: 30871
      protocol: TCP
    - name: bridge-api
      port: 8772
      targetPort: 8772
      nodePort: 30872
      protocol: TCP
EOF

    echo "    → k8s/namespace.yaml"
    echo "    → k8s/configmap.yaml  (flashmq.conf + writer.yaml embedded)"
    echo "    → k8s/pvc.yaml"
    echo "    → k8s/deployment.yaml  (image: ${IMAGE_TAG})"
    echo "    → k8s/service.yaml    (NodePort 31883/31901/30880/30871/30872)"
}

8. Step 5 — Package tar.gz

# Final packaging — $BUILD_DIR is cleaned by the EXIT trap
tar -czf "$OUT" -C "$BUILD_DIR" evelyn-deploy/

echo
echo "=== Done ==="
echo "    $OUT"
echo "    $(du -sh "$OUT" | cut -f1)  $(tar -tzf "$OUT" | wc -l) files"
echo
echo "  Inspect:  tar -tzf $OUT_NAME"
echo "  Extract:  tar -xzf $OUT_NAME"
echo "  Deploy:   see evelyn-deploy/k8s/ and evelyn-deploy/html/writer/evelyn_writer_project_build.html"

The trap ... EXIT at the top of the script deletes $BUILD_DIR automatically on exit — whether the build succeeds or fails.

9. Kubernetes Manifests (detail)

Apply order

Apply in this order — namespace must exist before other resources reference it:

kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/pvc.yaml
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

Or all at once:

kubectl apply -f k8s/

Port map (NodePort)

Service nameContainer portNodePortProtocol / Use
mqtt188331883TCP — MQTT (EMS hardware, bridge)
mqtt-ws900131901TCP — MQTT over WebSocket (browser dashboards)
http-ui808030880TCP — nginx HTML dashboards
health877130871TCP — parquet_writer health endpoint
bridge-api877230872TCP — bridge_api_container.py management REST

ConfigMap strategy

The evelyn-config ConfigMap is used two ways:

  • As the defaults source: the init-config init container copies from /etc/evelyn-defaults/ (mounted from ConfigMap) into /etc/evelyn/ (emptyDir) only when the files are absent.
  • As the bootstrap values: once a site-specific bridge has been configured via POST /bridge, the live config in /etc/evelyn/ diverges from the ConfigMap — this is intentional. The ConfigMap represents factory defaults.
Config persistence caveat: The evelyn-config volume is emptyDir — it is lost when the pod restarts. For production, replace the emptyDir with a second PVC or a ConfigMap + mutation webhook so bridge changes survive pod restarts. The parquet data PVC is persistent.

10. Using the Tar

On the build machine (phil-dev / CI)

# Build everything and produce the tar
bash scripts/evelyn-k8s-build.sh

# Optionally specify the image tag embedded in deployment.yaml
EVELYN_IMAGE=registry.example.com/evelyn:1.2.3 bash scripts/evelyn-k8s-build.sh

Build the container image from the tar

The tar includes the full Dockerfile (Option C three-stage build). Extract and run:

tar -xzf dist/evelyn-deploy-YYYYMMDD.tar.gz
cd evelyn-deploy/

# Build the image — Docker will compile FlashMQ and parquet_writer inside the build
docker build -t evelyn:latest .

# Or use the pre-compiled binaries from bin/ to speed up CI (skip Stage 1 & 2)
# by providing them as build args — see Dockerfile COPY --from=writer-builder notes

Push to a registry

docker tag evelyn:latest registry.example.com/evelyn:1.2.3
docker push registry.example.com/evelyn:1.2.3
# Then update image: in k8s/deployment.yaml and re-apply

Deploy to Kubernetes

# First deploy
kubectl apply -f evelyn-deploy/k8s/

# Check rollout
kubectl -n evelyn rollout status deployment/evelyn

# Check logs
kubectl -n evelyn logs -f deployment/evelyn

# Health check
NODE_IP=$(kubectl get nodes -o jsonpath='{.items[0].status.addresses[0].address}')
curl http://${NODE_IP}:30871/health

Set bridge to EMS hardware

# Replace <NODE_IP> with the Kubernetes node address
curl -X POST http://<NODE_IP>:30872/bridge \
     -H 'Content-Type: application/json' \
     -d '{"address":"<EMS_IP>","port":1883,"topic":"unit/#"}'

Native (non-k8s) deploy from the same tar

For a bare-metal or VM install without Kubernetes, the tar also includes scripts/install.sh which sets up systemd units:

tar -xzf evelyn-deploy-YYYYMMDD.tar.gz
cd evelyn-deploy/
sudo bash scripts/install.sh --parquet-path /data/parquet-evelyn

11. Notes & Caveats

TopicDetail
FlashMQ binary size ~8–12 MB stripped. Building from source takes ~2–3 min on a 4-core machine. To cache across builds, push the FlashMQ binary to a separate Docker layer or store it in bin/flashmq-cache in the repo and skip Step 2 if already present.
parquet_writer binary size ~15–20 MB. Arrow/Parquet libs are linked dynamically — the runtime image must have matching library versions (libarrow1600 libparquet1600 on Ubuntu 24.04).
K8s config persistence Config volume is emptyDir by default — lost on pod restart. For production, use a second PVC (evelyn-config-pvc) or bake the site config into the image at CI time.
Single-writer design This deployment pattern is for one parquet_writer instance per container. For the 4-sim gx10 topology (slices A/B/C/D), deploy four separate Kubernetes Deployments with different site_id, MQTT topic prefix, and PVC.
Air-gapped build Step 2 (FlashMQ) requires internet access to clone from GitHub. For air-gapped environments, pre-clone FlashMQ into docker/evelyn-container/flashmq-src/ and modify the build step to mount and build from that path instead.
Arrow APT repo Ubuntu 24.04 ships Arrow 14 in the default apt repo. If a newer Arrow is required, add the Apache Arrow APT source inside the Docker build container before apt-get install.
bridge_api_container.py This file is generated / defined in evelyn_container_design.html and must be written to docker/evelyn-container/bridge_api_container.py in the repo before running the build script. The build script will fail with a clear error if it is absent.
Verifying the binary works before packaging: 
# Quick sanity check after Step 1 — should print usage/version and exit cleanly
docker run --rm \
  -v "$REPO_ROOT/source/parquet_writer_cpp/parquet_writer":/bin/parquet_writer:ro \
  ubuntu:24.04 bash -c "
    apt-get install -y -q libmosquitto1 libyaml-cpp0.8 libsimdjson0 \
      libarrow1600 libparquet1600 libsnappy1v5 libzstd1 liblz4-1
    /bin/parquet_writer --help
  "
