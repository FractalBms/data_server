Edge Parquet Architecture

Instead of streaming raw messages to AWS, each site controller runs the full NATS pipeline locally, generates compressed Parquet files, and uploads them directly to S3. AWS is reduced to S3 + a single shared Subscriber API instance. Live monitoring is served via a NATS Leaf Node — a single outbound TLS connection that forwards messages to the AWS NATS server in real time.

Site Controller (edge)

  • Full NATS stack: FlashMQ → Bridge → JetStream → Parquet Writer
  • Parquet Writer uploads to S3 every 60 seconds (primary path)
  • NATS Leaf Node forwards every message to AWS in real time (live path)
  • Continues operating fully offline — queues locally, uploads on recovery
  • NATS JetStream WAL on disk — survives power cycle, replays on restart

AWS (thin read layer)

  • NATS Server receives live messages via Leaf Node from all sites
  • Gap-fill Parquet Writer consumes from AWS NATS — writes to S3 only if site hasn't uploaded that window (S3 key check)
  • Subscriber API: history from S3, recent from in-memory buffer
  • One t3.small serves all 80 sites — cost doesn't scale with site count

Dual-path Resilience

  • Path 1 — Site Parquet Writer → S3 (every 60s, primary)
  • Path 2 — Leaf Node → AWS NATS → Gap-fill Writer → S3 (backup)
  • Power loss mid-buffer: messages already on AWS via Path 2 — not lost
  • WAN link drops: site continues, S3 already has data via Path 1
  • Power + WAN simultaneous: NATS WAL on local disk, replay on recovery
  • More resilient than current cloud pipeline on every failure mode

NATS Leaf Node Config

  • Single outbound TLS connection, configured in nats-server.conf
  • No code changes — purely infrastructure config
  • All sites appear on the same subject space at AWS
  • ~40× less data transfer than raw MQTT streaming
leafnodes {
  remotes [{
    url: "nats://aws:7422"
    credentials: "/etc/nats/site.creds"
  }]
}

MQTT Publishing — Generator & Stress Runner

Both publishers use paho-mqtt (QoS 1, keepalive 60 s) and connect to the local MQTT broker on the machine they run on. They share the cell_generator.py simulation core (Gaussian random-walk with drift reversal at bounds) to produce realistic-looking sensor trajectories.

Generator — phil-dev, on demand

  • Broker: localhost:1884 (Mosquitto — managed by process manager)
  • Topology: 2 sites × 3 racks × 4 modules × 6 cells = 144 cells
  • Rate: 1 publish loop per second → 144 msgs/s
  • Topic mode: per_cell (configurable; also supports per_cell_item)
    batteries/site=0/rack=1/module=2/cell=3
  • Payload (per_cell): measurements are nested — each field is an object: {"voltage": {"value": 3.71, "unit": "V"}, "current": {"value": -0.4, "unit": "A"}, …, "timestamp": 1741234567.89}
  • Controlled interactively via WebSocket :8765 (start / stop / update_config)

Stress Runner — spark-22b6, 24/7

  • Broker: localhost:1883 (FlashMQ — OS service)
  • Topology: 1 project × 3 sites × 4 racks × 8 modules × 12 cells = 1,152 cells
  • Rate: 1 loop per second per site task → ~1,152 msgs/s sustained
  • Topic: always includes project= segment
    batteries/project=0/site=0/rack=1/module=2/cell=3
  • Payload: measurements are flat top-level fields: {"timestamp": …, "project_id": 0, "site_id": 0, "rack_id": 1, …, "voltage": 3.71, "current": -0.4, …, "power": -1.48} — power is derived (voltage × current) and only present when both fields are in the schema
  • One asyncio task per (project, site) pair; single shared paho client in loop_start mode
  • Status broadcast every second via WebSocket :8769 (read-only stats)

FAQ — Configuration & Data Flow

Click a question to expand. Answers are grounded in the actual config files and source code.

What configuration does local FlashMQ need for incoming MQTT data? ›

FlashMQ is configured in orig_aws/flashmq/flashmq.conf. The minimum required blocks for the battery pipeline are:

listen {
    port     1883
    protocol mqtt
}

allow_anonymous  true          # no auth — add TLS + auth for production
log_file         /var/log/flashmq/flashmq.log
log_level        info

# High-throughput tuning
client_initial_buffer_size  65536    # 64 KB per-client receive buffer
max_packet_size             1048576  # 1 MB max MQTT packet
max_qos_msg_pending_per_client  1000 # QoS-1 in-flight queue per client
  • No persistence is configured — FlashMQ is intentionally stateless; durability is NATS JetStream's job downstream.
  • Three clients connect: the generator (publishes), the NATS bridge (subscribes batteries/#, QoS 1), and Telegraf in the orig_aws stack (also subscribes batteries/#, QoS 1).
  • On spark-22b6 FlashMQ runs as an OS systemd service, not via the process manager — it is always running before the pipeline starts.
  • To enable per-publish logging (useful when debugging): uncomment log_subscriptions true in the config, then systemctl reload flashmq.
How does the local FlashMQ connect to the AWS instance? ›

FlashMQ itself does not connect to AWS directly. Data leaves the machine via two separate downstream consumers that both subscribe to it:

  • NATS stack path: The nats_bridge (Python, aiomqtt) subscribes to FlashMQ on localhost:1883, converts each message to a NATS subject, and publishes it to the local NATS JetStream. The parquet_writer then drains that stream and uploads Parquet files to S3 — either MinIO locally (http://spark-22b6:9000) or real AWS S3 (remove endpoint_url from parquet_writer/config.yaml and supply real AWS credentials).
  • orig_aws path: Telegraf subscribes to FlashMQ (servers = ["tcp://flashmq:1883"] in Docker, or tcp://localhost:1883 natively) and writes to InfluxDB2, which can be a local container or an AWS EC2 instance. Change the Telegraf output URL from http://influxdb:8086 to http://<ec2-host>:8086 to point at a remote InfluxDB2.

There is no MQTT bridge configured (FlashMQ-to-FlashMQ forwarding). All AWS connectivity is in the consuming services, not in the broker.

How do we monitor traffic through the pipeline? ›

Several complementary views:

  • NATS HTTP API (nats.html): GET :8222/varz shows in_msgs, out_msgs, in_msgs_rate, out_msgs_rate, and slow consumers in real time. GET :8222/jsz?streams=true&consumer=true shows per-stream message counts and per-consumer num_pending (pipeline lag).
  • FlashMQ $SYS topics: FlashMQ publishes broker stats every 10 s to $SYS/broker/clients/connected, $SYS/broker/messages/publish/received, $SYS/broker/messages/publish/sent, and $SYS/broker/bytes/received. Subscribe with any MQTT client: mosquitto_sub -t '$SYS/#' -v.
  • Process manager (dashboard.html): Shows running/stopped state for all managed processes including nats, nats_bridge, parquet_writer.
  • Subscriber API /diag endpoint: GET :8768/diag returns a JSON snapshot combining NATS JetStream state (messages, bytes, consumer lag, fill %), recent S3 file list, and live subscriber stats. This is the single-call health check for the whole pipeline.
  • Parquet writer log: Logs every flush — row count, S3 key, and bytes uploaded — once per flush_interval_seconds (default 60 s).
What configuration connects the NATS bridge to FlashMQ and NATS? ›

All settings live in source/nats_bridge/config.spark.yaml (spark-22b6) or config.yaml (phil-dev):

mqtt:
  host:      localhost
  port:      1883          # FlashMQ port
  client_id: nats-bridge
  topic:     "batteries/#"  # wildcard — catches both topic shapes
  qos:       1              # at-least-once delivery
  username:  ""             # leave blank for allow_anonymous

nats:
  url:             "nats://localhost:4222"
  stream_name:     BATTERY_DATA
  stream_subjects:
    - "batteries.>"      # NATS wildcard — all batteries.* subjects
  max_age_hours:   48       # messages retained 48 h as replay buffer
  max_bytes:       # default 8 GB — overrides stream limit on each start

Topic conversion: The bridge replaces every / with . so MQTT hierarchy becomes NATS subject hierarchy:

MQTT:  batteries/project=0/site=0/rack=1/module=2/cell=3
NATS:  batteries.project=0.site=0.rack=1.module=2.cell=3

The bridge creates or updates the BATTERY_DATA JetStream stream on startup (idempotent). If NATS or MQTT is unavailable it retries with a fixed 2 s back-off loop — no manual restart required.

What configuration controls does NATS JetStream expose? ›

Server-level — manager/nats-server.conf:

port: 4222              # client / bridge / subscriber connections
http_port: 8222        # monitoring API (read-only)

jetstream {
  store_dir:        "./nats-data"   # on-disk message store location
  max_memory_store: 512MB
  max_file_store:   10GB
}

Stream-level — set by the bridge on startup via the NATS Go/Python API:

  • stream_name: BATTERY_DATA — single stream for all battery subjects
  • stream_subjects: ["batteries.>"] — captures all batteries.* NATS subjects
  • retention: LimitsPolicy — messages kept until age or size limit, then dropped from head
  • storage: FileStorage — persisted to nats-data/ on disk (survives restarts)
  • max_age_hours: 48 — configurable per-deploy in bridge config; messages older than this are evicted
  • max_bytes: 8 GB (default in bridge code, overrides previous stream config each restart)

Consumer-level — two consumers are created by pipeline services:

  • parquet-writer — durable pull consumer, deliver_policy: ALL, ack_wait: 60 s, max_deliver: 10
  • Subscriber API live sub — ephemeral push consumer, deliver_policy: NEW, auto-deleted when it unsubscribes
How does NATS handle queries from the monitor / subscriber systems? ›

NATS itself only delivers messages — it does not run queries. The Subscriber API wraps NATS to provide two query modes:

Live stream (push consumer):

  • When a browser sends {"type": "subscribe", "subject": "batteries.>"} over WebSocket to :8767, the Subscriber API creates an ephemeral NATS push consumer with deliver_policy: NEW.
  • Each incoming NATS message is JSON-decoded, nested {value, unit} objects are flattened to scalars, and the result is forwarded to all connected WebSocket clients as {"type": "live", "subject": "…", "payload": {…}}.
  • The subscription auto-rebuilds on NATS reconnect.

History queries (DuckDB over S3):

  • When a browser sends {"type": "query_history", "query_id": "q1", "site_id": "0", "from_ts": …, "to_ts": …, "limit": 1000}, the server translates this into a DuckDB SQL query against the Parquet files in S3.
  • DuckDB uses the httpfs extension to read s3://battery-data/project=*/…/*.parquet with hive partitioning — it prunes to only the relevant project= and site= directories.
  • Results come back as {"type": "history", "query_id": "q1", "columns": […], "rows": […], "total": N, "elapsed_ms": N}.
  • NATS is not involved in history queries at all — they go directly to S3/MinIO.

NATS monitoring API (nats.html):

  • Browser makes plain GET requests directly to :8222/varz, /jsz, /connz etc. — this is a built-in NATS feature, no custom code required.
How is MQTT data turned into Parquet files, and which config items control it? ›

The Parquet Writer (source/parquet_writer/writer.py) runs as a durable NATS pull consumer. The pipeline is:

  1. Connect to NATS :4222 and register durable consumer parquet-writer on stream BATTERY_DATA, subject filter batteries.>.
  2. Pull messages in batches of fetch_batch (default 500) with a 1 s timeout — fills an in-memory buffer of (payload_dict, nats_msg) tuples.
  3. Flush the buffer when either flush_interval_seconds (default 60 s) elapses or the buffer reaches max_messages (default 10 000).
  4. On flush: JSON-decode each message; flatten nested {value, unit} objects to scalars; coerce id columns to int64 and measurement columns to float64; add project_id = 0 if absent.
  5. Group rows by (project_id, site_id) — one Parquet file per group per flush.
  6. Build the S3 key from the partitions template (default: project={p}/site={s}/{year}/{month}/{day}/{timestamp}.parquet).
  7. Upload to S3/MinIO with up to 5 exponential-backoff retries.
  8. Acknowledge NATS messages only after successful upload — if the writer crashes mid-flush, NATS replays the unacknowledged batch on restart.

Key config items in source/parquet_writer/config.yaml:

nats:
  url:           nats://localhost:4222
  stream_name:   BATTERY_DATA
  consumer_name: parquet-writer   # durable — survives restarts
  subject_filter: "batteries.>"

buffer:
  flush_interval_seconds: 60   # max age of any buffered message before flush
  max_messages:           10000 # flush early if buffer fills first
  fetch_batch:            500  # NATS pull batch size per iteration

parquet:
  compression: snappy          # snappy | zstd | gzip

s3:
  endpoint_url: http://spark-22b6:9000  # remove for real AWS S3
  bucket:       battery-data
  access_key:   minioadmin
  secret_key:   minioadmin
  project_id:   0
  partitions:
    - "project={project_id}"
    - "site={site_id}"
    - "{year}"
    - "{month}"
    - "{day}"

The partitions list is the full list of path segments in order — reorder or add {hour} to change granularity. Available variables: {project_id}, {site_id}, {year}, {month}, {day}, {hour}.

How does monitor.html fetch data — and is the path the same for NATS and InfluxDB2? ›

monitor.html implements two completely separate data paths selected by the stack-selector buttons, but uses identical rendering code for both:

NATS stack (Subscriber API — :8767)

  • Opens a WebSocket to ws://<host>:8767.
  • On connect sends {"type": "subscribe", "subject": "batteries.>"} — live messages arrive as {"type": "live", "payload": {…}} events; these update the heatmap and time-series charts in real time with no polling.
  • For history (time-series chart data) sends {"type": "query_history", …}; the server responds with a columnar {"type": "history", "columns": […], "rows": [[…],…]} result that the page maps to chart data arrays.

orig_aws stack (InfluxDB2 — :8086)

  • No WebSocket — uses plain fetch() HTTP POST to /api/v2/query?org=battery-org with header Authorization: Token battery-token-secret and Content-Type: application/vnd.flux.
  • Three separate Flux queries are fired in parallel on each refresh: timeseries (aggregateWindow mean), heatmap (last voltage per module/cell), and writerate (count of rows in last 30 s). The responses are InfluxDB2 annotated CSV, parsed by a custom CSV parser in the page.
  • Auto-refreshes on a configurable timer (3/5/10/30 s) because InfluxDB2 has no push mechanism.

Using the Flux-compat shim instead of real InfluxDB2

  • Point the "Host" field at the Subscriber API host and set port to 8768 instead of 8086. The shim at :8768 accepts the same Flux POST and returns the same annotated CSV format — monitor.html requires no code changes.
  • This lets the orig_aws HTML stack be demonstrated even when no InfluxDB2 is running, by reading from the same S3 Parquet files that the NATS stack writes.

What both paths share

  • Same chart rendering functions (drawLineChart, renderHeatmap, renderMiniStats).
  • Same cell selector (site / rack / module / cell dropdowns).
  • Same host saved to localStorage keys ds_host (phil-dev) and aws_host (spark-22b6).
What is the Edge Parquet architecture and why does it reduce AWS costs so dramatically? ›

In the current cloud pipeline, every raw MQTT message is streamed to AWS for processing — at 8,000 msgs/s that's ~8 MB/s, ~700 GB/day per site, plus per-message IoT Core and Lambda charges.

The edge architecture moves Parquet file generation to the site controller. Instead of streaming raw messages, each site uploads pre-compressed Parquet files every 60 seconds — the same data compressed to ~12 MB/flush (~17 GB/day), roughly 40× less transfer. IoT Core and Lambda are eliminated entirely.

Why the ratio widens at scale:

  • InfluxDB2 must scale vertically — its cardinality index grows with series count (site × rack × module × cell × measurement), forcing progressively larger r5 instances.
  • The NATS/Parquet stack scales horizontally — NATS handles millions of msgs/s on a single node; S3 scales with data volume at a fixed $0.023/GB.
  • The Subscriber API is stateless — one t3.small on AWS serves all 80 sites. EC2 cost is flat regardless of site count.

Result: 80 sites at ~$860/month vs ~$76,200/month with InfluxDB.

What happens to the last 60 seconds of data if the site controller loses power? ›

The 60-second buffer in the Parquet Writer is in RAM — a hard power cut before the flush would normally lose it. However, the Edge Parquet architecture provides two independent protections:

Protection 1 — NATS Leaf Node (Path 2)

  • The NATS Leaf Node forwards every message to AWS in real time — well before the 60-second flush.
  • On power loss, those messages are already sitting in AWS NATS JetStream.
  • The AWS gap-fill Parquet Writer detects the missing S3 window and writes it from the AWS NATS data.
  • No data loss — the gap-fill writer activates automatically, no manual intervention.

Protection 2 — NATS JetStream WAL on disk

  • NATS JetStream writes every message to a write-ahead log on local disk before acknowledging to the bridge.
  • The Parquet Writer only acknowledges messages to NATS after a successful S3 upload.
  • On restart, NATS replays all unacknowledged messages from the last committed position.
  • The writer uploads the missing window to S3 on its first flush after restart.

Remaining risks and mitigations:

  • NATS disk corruption: use a journaling filesystem (ext4/xfs — Linux default). Optionally add sync: always in nats-server.conf for fsync-per-message durability (~20% throughput cost).
  • MQTT → NATS bridge window (~milliseconds): devices using QoS 1 retransmit automatically if the broker didn't ack.
  • Simultaneous power loss + WAN failure: covered by the NATS WAL — replays to S3 when both recover.
  • Best practice: a small UPS ($50–100) gives a 30-second clean shutdown — NATS flushes its WAL and the writer acks cleanly. With a UPS the power cycle risk is effectively zero.

This makes the edge architecture more resilient than the current cloud pipeline for every failure mode — the InfluxDB stack has no equivalent recovery mechanism if IoT Core drops messages during a WAN outage.

Why does NATS reduce AWS costs so dramatically? ›

The old stack charged AWS per message at high volume. The NATS stack charges AWS per byte stored at much lower volume.

The root cause of the $76k bill

The InfluxDB stack pushes every raw MQTT message through AWS IoT Core, then triggers a Lambda invocation to write it to InfluxDB. At stress load: 

8,000 msg/s × 3,600s × 24h × 30d = ~20 billion messages/month per site
× 80 sites = ~1.7 trillion messages/month

IoT Core:       $0.30 / million messages  = ~$3,000/site/month
Lambda:         per-invocation cost       = ~$2,000/site/month
Data transfer:  8 MB/s raw × 80 sites    = ~$3,600/site/month

Every single cell reading hits AWS as an individual HTTP transaction. AWS bills you for each one.

What NATS changes

The NATS bridge moves the pipeline so that raw messages never reach AWS at all. Instead:

  • Messages accumulate in local NATS JetStream for 60 seconds.
  • The parquet writer batches them, compresses with Snappy, and uploads one file per site per minute.
  • That one file replaces ~480,000 individual messages (8,000 msg/s × 60 s).
Old:  8,000 msg/s × ~1 KB  =  8 MB/s  =  ~700 GB/day per site
New:  one 60 s parquet file ≈ 12 MB compressed  =  ~17 GB/day per site
                                                      ↑ 40× less transfer

Because data goes direct to S3 — not through IoT Core or Lambda — the per-message charges disappear entirely:

  • IoT Core: ~$3,000/site/mo → $0
  • Lambda: ~$2,000/site/mo → $0
  • Data transfer: ~$3,600/site/mo → ~$0.35/site/mo
  • EC2: $182/site/mo → $10 shared
  • S3: $0 → $10/site/mo
  • Total: ~$226/site/mo → ~$21/site/mo. Saving across 80 sites: ~$75,000/month.

The deeper principle

IoT Core + Lambda is an event-driven serverless pattern designed for sporadic, low-volume device messages — think a thermostat reporting once a minute. Battery telemetry at 8,000 msg/s is an industrial time-series workload, and AWS pricing for that pattern is punishing at scale.

NATS acts as a local aggregation layer that converts a firehose of small events into a trickle of large files before touching AWS. S3 is extremely cheap for bulk storage; IoT Core is extremely expensive for bulk messaging. The architecture simply stops using the expensive service.

What does NATS do? ›

NATS is a lightweight, high-performance message broker — a central bus that lets processes send messages to each other without knowing about each other directly. A publisher sends a message to a subject; any number of subscribers listening on that subject receive it. The publisher and subscriber never connect directly and don't need to be running at the same time.

Core concepts

  • Subject — a dot-separated address string, e.g. batteries.site=0.rack=1.module=2.cell=3. Wildcards: * matches one token, > matches everything remaining. batteries.> catches all cell messages from all sites.
  • Publish / Subscribe — fire-and-forget. If no subscriber is listening the message is dropped. Fast and stateless; suitable for live dashboards where losing the occasional message is acceptable.
  • Queue groups — multiple subscribers in the same group share the load; each message goes to exactly one of them. Used to horizontally scale consumers.
  • Request / Reply — built-in RPC pattern. Publisher includes a reply subject; subscriber responds to it. Not used in this pipeline but useful for control commands.

JetStream — the persistence layer

Plain NATS pub/sub is lossy. JetStream adds durable, disk-backed message storage on top of the same subject space. This is what this pipeline uses exclusively.

  • Stream — captures all messages matching a subject filter and stores them on disk. Here: stream BATTERY_DATA captures batteries.>, retaining up to 10 GB.
  • Consumer — a named cursor into a stream. Tracks which messages have been acknowledged so that on restart the process replays only what it missed. Two types used here:
    • Pull consumer (Parquet Writer, gap-fill writer) — fetches batches on demand, acks only after S3 upload succeeds.
    • Push consumer (Subscriber API) — server pushes new messages as they arrive; used for the live WebSocket feed.
  • DeliverPolicy — controls where a new consumer starts: ALL replays from the beginning of the stream; NEW delivers only messages arriving after the subscription is created.
  • AckPolicy — unacknowledged messages are redelivered after ack_wait (60 s here). If the writer crashes mid-upload, NATS retries delivery automatically on restart.

Leaf nodes

A leaf node is a NATS server that connects outbound to a hub NATS server over a single TLS connection and forwards a filtered set of subjects. In this architecture each site NATS connects to the AWS NATS as a leaf node, giving AWS a real-time copy of all battery messages with no application-level code involved.

Why NATS instead of Kafka, RabbitMQ, or MQTT alone?

  • Single ~20 MB binary, no JVM, no ZooKeeper, no external dependencies.
  • Handles millions of messages per second on modest hardware.
  • JetStream WAL gives zero-loss guarantees without a separate queue service.
  • Leaf nodes make multi-site fan-in to a central hub trivial — one config block, no code.
  • MQTT is device-friendly but has no durable consumer model suitable for batch Parquet writing. NATS JetStream fills that gap cleanly.
Can NATS and the bridge clean up or transform the MQTT topic format? ›

Yes, straightforwardly. The bridge is the ideal place because it already touches every message — adding transformation there keeps it in one process with no extra hop.

What "clean up" could mean

The current topic arriving at the bridge is:

batteries/site=0/rack=1/module=2/cell=3

converted as-is to NATS subject:

batteries.site=0.rack=1.module=2.cell=3

Two useful transformations:

  • Cleaner subject — strip the key names, keep only values: batteries.0.1.2.3. Consumers use positional wildcards (batteries.*.*.*.>). Shorter and faster to route, but less self-describing.
  • Payload enrichment — parse the topic and inject the hierarchy fields into the JSON body so downstream consumers never need to parse the subject at all: {"site_id": 0, "rack_id": 1, "module_id": 2, "cell_id": 3, "voltage": {…}}. This is what the parquet writer already expects — if a real device omitted those fields, the bridge could inject them from the topic.

How you would do it

A small change to bridge.py. Currently mqtt_topic_to_nats_subject is a one-liner. Extending the message loop to parse and enrich: 

import re

_TOPIC_RE = re.compile(
    r"batteries/site=(\d+)/rack=(\d+)/module=(\d+)/cell=(\d+)"
)

def parse_topic(topic):
    m = _TOPIC_RE.match(topic)
    if m:
        return {k: int(v) for k, v in zip(
            ("site_id","rack_id","module_id","cell_id"), m.groups()
        )}
    return {}

# In the message loop:
subject = f"batteries.{site}.{rack}.{module}.{cell}"  # clean subject
payload = json.loads(message.payload)
payload.update(parse_topic(str(message.topic)))        # inject ids if missing
await js.publish(subject, json.dumps(payload).encode())

NATS server-side option (JetStream transforms, 2.10+)

NATS can rewrite subjects server-side via subject_transform in the stream config, but it only does token-level remapping — not regex parsing of key=value strings. It cannot strip the key names. The bridge approach is the right tool here.

Trade-offs

  • Clean subject only — shorter routing keys, smaller wire overhead; positional so loses self-description.
  • Payload enrichment only — downstream code simpler, no subject parsing needed; subject stays verbose.
  • Both — maximum flexibility; bridge code slightly more complex.

For this pipeline payload enrichment is the higher-value change — the parquet writer and DuckDB queries both work on payload fields, not subject tokens. The subject format matters less once data is in Parquet.

What types of data can NATS handle, and does it offer translation? ›

NATS is payload-agnostic — it treats every message as raw bytes and does zero inspection or transformation of the content. Translation is not a NATS concept; that is left to the application layer.

What NATS handles

Any byte sequence: JSON, Protobuf, MessagePack, Avro, raw binary, plain text, compressed blobs. The only constraint is the default max message size (1 MB, configurable up to 64 MB via max_payload in nats-server.conf).

In this codebase the bridge publishes raw MQTT payloads straight into NATS with no transformation: 

# bridge.py
await js.publish(subject, message.payload)   # bytes — untouched

The parquet writer is where the bytes get interpreted — json.loads(msg.data) then _flatten().

What NATS does offer that is relevant

  • Subject-based routing — the subject hierarchy (batteries.site=0.rack=1…) is the closest thing to translation. Consumers subscribe to wildcards (batteries.>, batteries.site=0.>) to filter without touching payloads.
  • JetStream subject transforms (NATS 2.10+) — the server can rewrite subjects on ingress/egress via subject_transform in stream config. Useful for fan-out or renaming, but still payload-blind.
  • Header support — messages can carry key/value headers (like HTTP). You could tag messages with schema version, source site, or content-type without touching the payload. Not currently used here.
  • Leaf node subject filtering — the leaf node config restricts which subjects get forwarded to AWS (e.g. only batteries.>, not internal admin subjects).

What NATS explicitly does not do

  • No schema registry
  • No payload serialisation or deserialisation
  • No protocol translation (e.g. MQTT → AMQP)
  • No field extraction or filtering within a payload

The NATS Bridge pattern — what bridge.py is — is the standard answer to "how do I get data from protocol X into NATS": write a small process that speaks both. That is also why things like Telegraf, Kafka Connect, and NATS's own nats-kafka bridge exist.

Practical implication for this system

If the payload format ever changed (e.g. JSON → Protobuf for bandwidth), NATS would not need to change at all — update the bridge to encode and the parquet writer to decode. The NATS layer is completely insulated from serialisation choices.

How does MQTT data get to the local NATS server and to AWS? ›

Two separate mechanisms handle the two hops — one is bespoke code, the other is pure NATS config.

Hop 1 — MQTT → Local NATS (the bridge)

source/nats_bridge/bridge.py is a Python process that glues the two protocols together:

  • Subscribes to FlashMQ on batteries/#, QoS 1 (port 1884) via aiomqtt.
  • For every message, converts the topic separator — / → .:
    batteries/site=0/rack=1/module=2/cell=3
    → batteries.site=0.rack=1.module=2.cell=3
  • Calls js.publish(subject, payload) — waits for NATS server ack before moving on.
  • NATS stores the message durably to disk (WAL, up to 10 GB) before acking back to the bridge.
  • QoS 1 on the MQTT side means FlashMQ retransmits if the bridge disappears — no gap at this hop.
  • Both the MQTT and NATS connections retry with infinite back-off on failure.

Hop 2 — Local NATS → AWS (leaf node)

This is zero code — it is a NATS server feature configured in nats-server.conf. The site NATS server opens a single outbound TLS connection to the AWS NATS server and forwards every matching message automatically: 

leafnodes {
  remotes [
    {
      url: "nats://aws-nats.example.com:7422"
      credentials: "/etc/nats/site.creds"
      account: "BATTERY_DATA"
    }
  ]
}

AWS NATS receives messages as if they were published locally — the Subscriber API and gap-fill writer see no difference. The current manager/nats-server.conf does not yet contain this block; adding it is the one remaining config change needed to complete the edge architecture.

Combined flow:

Battery device
    │ MQTT QoS 1  (retransmits until FlashMQ acks)
    ▼
FlashMQ :1884
    │ MQTT subscribe batteries/#  (bridge.py / aiomqtt)
    ▼
bridge.py  ── topic / → .  ──►  js.publish()
    │
    ▼

    │

Design Principles

Self-contained pages. Each demo is a single HTML file with no build step, no framework, and no external CDN dependency. Open the file directly in a browser pointed at a running server.

Two stacks, one page set. monitor.html and dashboard.html support both the NATS stack and the orig_aws InfluxDB2 stack via the same localStorage host keys (ds_host and aws_host). Switching between stacks is a host toggle, not a separate page.

Transparency. api_test.html shows the exact bytes sent and received over WebSocket and HTTP. There is no abstraction hiding the wire format from the developer.

Progressive disclosure. The five-step workflow (Start → Generate → Monitor → Query → Inspect) introduces one pipeline stage per page. A newcomer can follow the sequence and observe data flowing through every stage without reading source code.

Always in sync. Demo pages connect to the live server so request/response examples always reflect the currently deployed API — documentation cannot drift from reality.
