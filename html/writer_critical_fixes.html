C1 Signal handler calls notify_all on condition_variable — undefined behaviour lines 917–920 ✓ Applied

condition_variable::notify_all() acquires an internal mutex. If the signal fires while g_mutex is held by another thread, the signal handler deadlocks. Under POSIX only a small set of functions (write, sem_post, _exit…) are async-signal-safe.

Fix: self-pipe pattern — write one byte to a pipe from the handler; the flush thread polls the pipe instead of blocking on the condvar.

Before — line 917
static void handle_signal(int) {
    g_shutdown = true;
    g_flush_cv.notify_all(); // NOT safe
}
After
// globals:
static int g_signal_pipe[2] = {-1,-1};

static void handle_signal(int) {
    g_shutdown.store(true,
        std::memory_order_relaxed);
    char b = 1;
    (void)write(g_signal_pipe[1], &b, 1);
}
Before — flush_thread_fn wait
g_flush_cv.wait_for(lock,
    std::chrono::seconds(
        g_cfg->flush_interval_seconds));
After — poll the pipe
lock.unlock();
struct pollfd pfd =
    { g_signal_pipe[0], POLLIN, 0 };
poll(&pfd, 1,
    g_cfg->flush_interval_seconds * 1000);
lock.lock();
In main() — before mosquitto_lib_init()
pipe(g_signal_pipe);  // or pipe2(..., O_CLOEXEC) on Linux

C4 on_message holds g_mutex for every message at 80k msg/sec — blocks MQTT thread lines 878–900

The MQTT network thread calls on_message which locks g_mutex to insert into g_buffers and update g_current_state. The flush thread holds the same lock while copying g_current_state — up to 10k std::map entries with heap-allocated keys. During that O(N) copy every incoming message queues behind it. Three options in increasing order of effort:

Option A — Chosen ✓ Eliminate the g_current_state copy — shared_ptr move at flush time Applied

The bottleneck is not the mutex itself but the O(N) map copy under the lock. Change g_current_state to a std::shared_ptr<StateMap>. The flush thread does std::move to take ownership and installs a fresh empty map — O(1) pointer swap under the lock. on_message writes through the pointer as before. Lock hold time in the flush thread drops from O(N) to O(1).

  • ✓ Minimal structural change — g_buffers swap loop untouched
  • ✓ Safe to land immediately alongside C1–C6
  • ✗ on_message still runs on the mosquitto network thread — Python subscriber gets no benefit
Before — O(N) copy under lock
static std::map<SensorKey,Row>
    g_current_state;

// flush_thread_fn — holds g_mutex
// for the entire map copy:
auto cs_snap = g_current_state;
// ↑ heap-allocates N entries
//   while MQTT thread is blocked
After — O(1) pointer move under lock
using StateMap = std::map<SensorKey,Row>;
static std::shared_ptr<StateMap>
    g_current_state{
        std::make_shared<StateMap>()};

// flush_thread_fn — O(1) move:
auto cs_snap = std::move(g_current_state);
g_current_state = std::make_shared<StateMap>();
// lock released immediately after
Option B Lock-free SPSC ring in writer.cpp — dedicated ingest thread ~1 week

on_message pushes to a lock-free single-producer single-consumer ring queue (mosquitto uses one network thread — true SPSC). A dedicated ingest thread drains the queue, batches rows into g_buffers, and updates state — so the flush lock is never contended from the network thread at all.

  • ✓ MQTT network thread never blocks — keepalive risk eliminated
  • ✓ Ring queue entries can be fixed-size (no heap per message)
  • ✗ Row struct still uses std::map<string,…> — heap allocations inside the ring entries
  • ✗ Python subscriber_api still parses MQTT on its own event loop
Option C — Future Shared memory ring buffer — writer produces, Python consumes via numpy.frombuffer subscriber_api phase

An mmap-backed SPSC ring. Schema is stored in the ring header at startup (field names + types from config or first message — content-agnostic, not hardcoded). Each row is fixed-size, cache-line aligned, no heap. Python attaches the same /dev/shm region and reads via numpy.frombuffer — zero copy, no GIL, no MQTT parsing in Python.

Ring layout (generic — field names from config)
RingHeader [256 bytes, two cache lines]
  write_seq  atomic<uint64>    // producer cursor
  _pad       [56 bytes]
  read_seq   atomic<uint64>    // consumer cursor
  _pad       [56 bytes]
  capacity   uint32
  row_bytes  uint16            // fixed at startup
  n_int_fields, n_float_fields uint8
  field_names[32][16]          // generic field registry

RingRow [fixed, aligned to 8 bytes]
  ts_ns      int64
  ints[]     int64  × n_int_fields
  floats[]   double × n_float_fields
Python — build dtype from header, zero-copy read
def dtype_from_header(hdr):
    fields = [('ts_ns', np.int64)]
    for i in range(hdr.n_int_fields):
        fields.append((hdr.field_names[i], np.int64))
    for i in range(hdr.n_float_fields):
        fields.append((hdr.field_names[hdr.n_int_fields+i], np.float64))
    return np.dtype(fields)

ring = np.frombuffer(mm, dtype=dtype_from_header(hdr), offset=HEADER_SIZE)
  • ✓ Solves writer C4 and subscriber_api M1 simultaneously
  • ✓ mmap-backed — ring survives writer crash, replay on restart (replaces WAL for live window)
  • ✓ Content-agnostic — works for any MQTT payload shape
  • ✗ Row struct (heap maps) must be replaced — touches on_message, flush_partition, write_wal_ipc, WAL replay
  • ✗ Schema must be fixed at startup — dynamic field discovery needs a two-pass init

Recommended path: Land Option A now (eliminates the O(N) copy bottleneck, one day's work). Build Option C as a separate feature targeting subscriber_api's Python ingest problem — fold writer C4 in when the ring is proven.

C6 WAL replay reads only chunk(0) — silently drops rows in multi-chunk tables lines 537–541 ✓ Applied

Arrow ChunkedArray can hold multiple chunks when a table is assembled from multiple RecordBatches via Table::FromRecordBatches. chunk(0) silently discards everything after the first chunk. The fix is arrow::Concatenate which is a no-op for single-chunk columns.

Before — line 537
auto chunk0 = table->column(c)->chunk(0);
// only first chunk — rest silently dropped
if (tid == arrow::Type::INT64)
    int_cols.emplace_back(name,
        std::static_pointer_cast<
            arrow::Int64Array>(chunk0));
After — Concatenate all chunks
auto combined =
    arrow::Concatenate(
        table->column(c)->chunks());
if (!combined.ok()) continue;
auto arr = combined.ValueOrDie();
if (tid == arrow::Type::INT64)
    int_cols.emplace_back(name,
        std::static_pointer_cast<
            arrow::Int64Array>(arr));

arrow::Concatenate is in <arrow/compute/api.h> — already transitively included via <arrow/api.h>. Zero overhead for single-chunk arrays.

M9 Wrong shutdown order — mosquitto_disconnect after loop_stop, DISCONNECT never sent lines 1044–1046

mosquitto_loop_stop(force=true) kills the network thread before mosquitto_disconnect is called. The broker never receives a DISCONNECT packet and retains the session — QoS state, will messages, and subscriptions persist until the broker-side keepalive timeout.

Before
flush_thread.join();
mosquitto_loop_stop(g_mosq, true); // kills thread
mosquitto_disconnect(g_mosq);      // ← packet never sent
After — disconnect first, then stop
flush_thread.join();
mosquitto_disconnect(g_mosq);      // sends DISCONNECT
// give the network thread time to flush the packet:
std::this_thread::sleep_for(
    std::chrono::milliseconds(100));
mosquitto_loop_stop(g_mosq, false); // clean stop

This also interacts with M11 — see M11 for the combined correct shutdown sequence.

M10 g_mosq plain pointer — data race between main thread write and network thread reads lines 1005, 1038

The main thread writes g_mosq at startup and during reconnect; mosquitto's network thread reads it in callbacks. Under the C++ memory model this is a data race — formally undefined behaviour, practically a torn pointer read on 32-bit ARM.

Before
static struct mosquitto* g_mosq = nullptr;
After
static std::atomic<struct mosquitto*>
    g_mosq{nullptr};

All g_mosq reads become g_mosq.load() — a trivial mechanical change throughout. Callbacks that receive g_mosq as a parameter are unaffected.

M11 Final flush runs before mosquitto_loop_stop — guaranteed data loss window on shutdown lines 1041–1044

The flush thread exits when it sees g_shutdown. mosquitto_loop_stop is called after the final flush. Any messages delivered between the flush thread's last drain and loop_stop are silently discarded. This happens on every graceful shutdown.

Before — wrong order
g_shutdown = true;
flush_thread.join();     // final flush here
mosquitto_loop_stop(...);// messages still arrive
mosquitto_disconnect(...);
After — stop ingest first
g_shutdown.store(true);
mosquitto_disconnect(g_mosq); // signal broker
mosquitto_loop_stop(g_mosq, false); // drain network
// NOW flush — catches everything:
flush_thread.join();          // final flush

Combining M9 + M11: the correct shutdown sequence is set g_shutdown → disconnect → loop_stop → join flush thread.

M12 No config validation — bad values cause spin-loops, silent drops, or wrong compression lines 117–168

flush_interval_seconds <= 0 spins the flush thread at 100% CPU. max_total_buffer_rows <= 0 drops every single message. An invalid compression string falls through silently to Snappy.

Fix — add validate_config() called from main()
static bool validate_config(const Config& cfg) {
    bool ok = true;
    if (cfg.flush_interval_seconds <= 0) {
        std::cerr << "[config] flush_interval_seconds must be > 0\n"; ok = false;
    }
    if (cfg.max_total_buffer_rows <= 0) {
        std::cerr << "[config] max_total_buffer_rows must be > 0\n"; ok = false;
    }
    static const std::set<std::string> valid_compression{
        "snappy","gzip","brotli","lz4","zstd","none"};
    if (!valid_compression.count(cfg.compression)) {
        std::cerr << "[config] unknown compression: " << cfg.compression << "\n"; ok = false;
    }
    return ok;
}

M14 Integer scalars silently promoted to double — precision loss and duplicate Parquet columns lines 289–292

In parse_payload, flat JSON integers not in the configured int_cols list fall through to the float path. Values > 253 lose precision. Worse: if a JSON field name matches a topic-derived int column, the same field appears in both row.ints and row.floats, producing two Parquet columns with the same name.

Before — parse_payload float fallback
// integer field not in int_cols:

// ← may duplicate row.ints[key]
After — skip if already in ints

    row.floats[key] = (double)int_val;
// else: already stored as int64 — skip

For the precision issue: widen int_cols config or add a configurable threshold; values above 253 that must round-trip exactly should always be in int_cols.
