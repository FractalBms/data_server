Observations & Replay Notes

site_id not in columns. The site ID 02285AD0 is embedded in the filename prefix only. Any SQL query over the parquet glob must use a filename-scoped glob (e.g. 02285AD0_*.parquet) rather than a WHERE site_id = ... clause.
PCS outlier value. pcs device has MAX(value) = 4,294,967,296 (= 2³², likely an unhandled overflow/sentinel). Check these rows before replay to avoid corrupting target systems.
Replay feasibility. At ~86M rows/hour = 23,900 msg/s, real-time replay requires a broker (FlashMQ) and subscriber stack capable of sustaining that throughput. The data has full topic reconstruction columns available: unit_id, device, instance, point_name. No mqtt_topic column is present — topics must be reconstructed. Root pattern: ems/site/{site_id}/unit/{unit_id}/{device}/{instance}/{point_name}/float.
Project ID. All rows carry project_id = 0. This column is currently unused for partitioning. It could be repurposed as a tenant ID in multi-project deployments.
BMS dominant load. bms signals account for 84.1% of total rows. The ~2,392 distinct bms point names per unit (× 12 units) suggest a full polling sweep at fixed intervals. Fault/warning flags dominate by name count but not by unique value — high deduplication potential for COV filtering.
Wide-format compaction RAM requirement. A 3-hour window contains ~258M long rows. Pivoting to wide format creates ~3.1M rows × 3,894 columns. DuckDB materialises the full pivot hash table in memory before writing: estimated peak usage 20–40 GB RAM per window (sparse float32 matrix + hash overhead). Machines with less than 32 GB available RAM should process one window at a time and close the connection between windows. The 8 full windows would require sequential processing; do not attempt parallel wide compaction unless >64 GB RAM is free.

For an exact replay, the DevOps team should re-capture a short session with store_mqtt_topic: true to record the full original topic per row, eliminating all ambiguity.
